# Design: Mixed-Chain Benchmark Suite (MixKinBench)

## Context

用户已开发新的数值逆运动学(IK)求解?目标是验证其在各种机械臂配置下的性能和鲁棒性。现有基准测试主要关注标准的 6-7 DOF 旋转关节机器?缺乏对高 DOF 和混合关节类型的系统评估?

混合关节系统(旋转 + 移动)引入了以下挑?
- **单位不一?*: 旋转关节使用弧度,移动关节使用?导致雅可比矩阵的梯度缩放不同
- **工作空间复杂?*: 混合链产生非标准工作空间流形
- **求解器敏感?*: 数值优化算法可能对不同关节类型表现不同

## Goals / Non-Goals

**Goals**:
- 提供程序化生成任?DOF (6-100) 混合关节串行链的能力
- 生成可验证的基准测试数据?使用 FK oracle 确保正确?
- 测量求解器在不同自由度和关节类型组合下的性能指标
- 支持冷启动、热启动、轨迹跟踪等多种 IK 场景
- 保持与现?Google Benchmark 框架的兼容?

**Non-Goals**:
- 不支持并联机器人或闭链机?
- 不进行碰撞检测或任务空间约束测试
- 不生成实时可视化(可选的后处理可视化工具)
- 不替换现有的标准机器人基准测?UR5, Panda)

## Decisions

### Decision 1: Python 作为数据生成工具

**选择**: 使用 Python + NumPy/Pinocchio 生成 URDF 和测试数据集

**理由**:
- Python 生态系统丰?适合科学计算和数据处?
- Pinocchio 提供可靠?FK oracle(或使?kinex 自身?Python 绑定)
- 便于生成随机种子、参数扫描、数据序列化
- 不影响核?C++ 库的依赖(仅作为开发工?

**Alternatives considered**:
- C++ 直接生成: 实现复杂,缺乏灵活?难以调试
- 手动创建 URDF: 无法扩展?100 DOF,维护成本?

### Decision 2: 随机关节类型分布

**选择**: 使用随机概率 $P_{prism} \approx 0.2 - 0.3$ 生成混合?

**理由**:
- 真实机器人通常以旋转关节为?少量移动关节
- 随机分布可覆盖多种拓扑结?
- 支持用户指定种子以实现可重复?

**Alternatives considered**:
- 固定模式 (R-P-R-P...): 缺乏多样?无法测试非对称拓?
- 全旋转或全移? 退化为特殊情况,不体现混合挑?

### Decision 3: DH 参数 vs URDF 格式

**选择**: 生成标准 URDF 格式

**理由**:
- kinex 库原生支?URDF 解析
- URDF 是机器人领域的工业标?便于与其他工?RViz, PyBullet)集成
- 支持完整的几何信?虽然基准测试不需要可视化网格)

**Alternatives considered**:
- DH 参数: 表达简洁但不通用,需要额外转换逻辑

### Decision 4: 基准测试指标

**选择的核心指?*:
1. **Success Rate** (%): $\|FK(q_{sol}) - T_{target}\| < \epsilon$
2. **Iterations per Solve**: 平均迭代次数
3. **Execution Time** (µs): 每次求解的平均时?
4. **Position Error** (mm): 成功解的平均位置误差
5. **Rotation Error** (deg): 成功解的平均旋转误差
6. **Joint Type Sensitivity**: 按旋?移动关节分组的误差分?

**理由**: 这些指标直接反映求解器的收敛性、精度和效率

### Decision 5: Ground Truth 生成策略

**选择**: 使用 FK Oracle 生成测试用例

**流程**:
1. 随机采样关节向量 $q_{gt} \in \mathbb{R}^N$ (在关节限制内)
2. 计算目标位姿 $T_{target} = FK(q_{gt})$
3. 生成不同的初始猜?$q_{init}$:
   - 冷启? $q_{init} = 0$ 或远?$q_{gt}$
   - 热启? $q_{init}$ 接近 $q_{gt}$
4. 存储测试用例 $(T_{target}, q_{init}, q_{gt})$

**理由**: 确保目标位姿是可达的,便于验证求解器的正确?

## Risks / Trade-offs

### Risk 1: 生成的机器人可能不现?

**缓解**: 
- 限制链节长度?[0.1, 0.5] ?
- 移动关节限制?[-0.2, 0.5] ?避免无限工作空间)
- 旋转轴交替使?X/Y/Z 以确保空间复杂?

### Risk 2: ?DOF (100+) 可能导致求解器失败或耗时过长

**缓解**: 
- 分层测试(10, 20, 50, 100 DOF),逐步增加复杂?
- 设置超时限制(?1 ?
- 使用热启动减少迭代次?
- 记录失败案例供后续分?

### Risk 3: Pinocchio 依赖增加环境配置复杂?

**缓解**: 
- ?Pinocchio 作为可选依?仅用于数据生?不影响运行时)
- 提供备选方?使用 kinex 自身?Python 绑定作为 FK oracle
- 预生成数据集并版本控?用户可直接使?

### Risk 4: 数据集文件可能过?

**缓解**: 
- 使用紧凑的二进制格式(?NumPy .npz)或压?JSON
- 提供小型、中型、大型数据集供不同使用场?
- 不将数据集纳入主仓库(放在 LFS 或外部下?
